Phase 3
Agent Profiles
1
ZepEntityReader filters entities by type from graph
2
LLM enriches each entity with Zep context retrieval (2× passes)
3
Generates OASIS profiles: persona, MBTI, profession, sentiment, activity level
4
LLM generates simulation config (time, events, agent activity)
OasisProfileGenerator · LLM
Phase 4
Simulation
1
Spawns Python subprocess running OASIS Twitter / Reddit simulation
2
Each round: agents observe feed → LLM decides action → execute (post, like, repost…)
3
Actions logged to JSONL; Zep graph updated with episode memory
4
IPC via filesystem — Flask ↔ subprocess for interview commands

Report & Interaction

2

4

Key Architectural Patterns

  • Async Tasks Background threads + TaskManager singleton; frontend polls status endpoints
  • IPC Filesystem-based command/response between Flask and simulation subprocess
  • ReACT Report agent uses Think → Act → Observe → Reflect loop with Zep tool calls
  • Persistence File-based JSON — no database; projects, simulations, reports on disk
  • LLM-First Every pipeline stage depends on LLM calls — ontology through reports

Architecture Observations

  • Zep SPOF Graph, search, and memory all route through Zep Cloud — no fallback
  • No job queue Raw threads with no retry, dead-letter, or horizontal scaling
  • FS persistence No DB means no querying, transactions, or concurrent-access safety
  • LLM cost Every stage makes LLM calls — costs and latency compound across the pipeline
  • FS-based IPC Polling files for inter-process communication adds latency and fragility
